💳 Flujo pago completo (Stripe → Verifactu → DocuSeal → portal) ›
  1. 1Cliente añade producto al carrito y va a Stripe Checkout — src/app/api/checkout/route.ts
  2. 2Stripe llama a webhook /api/webhooks/stripe con firma HMAC — src/app/api/webhooks/stripe/route.ts
  3. 3checkout.session.completed → orders.status = 'processing' + Verifactu crea factura — src/lib/verifactu/easyverifactu.ts
  4. 4Email PaymentConfirmation via Resend — src/emails/PaymentConfirmation.tsx
  5. 5Ops sube PDF borrador en /ops/pedido/[id] — src/app/ops/actions.ts
  6. 6DocuSeal recibe PDF → cliente firma → webhook /api/webhooks/docuseal (HMAC-SHA256) — src/app/api/webhooks/docuseal/route.ts
  7. 7Descarga PDF firmado → Supabase Storage bucket documents + hash_sha256_signed calculado
  8. 8revalidateTag('orders' | 'orders-${'$'}{uid}' | 'order-${'$'}{id}') → portal refresca en <5 s
  9. 9Email DocumentReady con signed URL 1 h — cliente descarga desde /portal/pedido/[id]

  4. 4Submit → orders.status = 'processing' → ops empieza redacción

P0 Switch Stripe test → live + smoke 1 € 45-60 min · siguiente›
  1. 0Pre-req crítico: Stripe account con charges_enabled: true y payouts_enabled: true. Sin esto, cualquier intento de checkout en live devuelve account_invalid antes del webhook. Ver sub-bloque "Activar charges_enabled" justo debajo.
  2. 0.aActivar charges_enabled en Stripe — Dashboard → Activate payments (barra lateral izquierda, o URL directa https://dashboard.stripe.com/account/onboarding). Completar 5 secciones hasta que todas marquen ✓:
    • Business details: tipo de entidad (autónomo / S.L. / asociación), nombre legal exacto (DNI/CIF match), tax ID (NIF o CIF), industria MCC (recomendado 8931 Accounting, Auditing, Bookkeeping Services o 8111 Legal Services), dirección fiscal.
    • Public details: nombre público "Afiladocs", support_email: soporte@afiladocs.com (o ops@ tras P1), support_phone, support_url: https://afiladocs.com/contacto, business_url: https://afiladocs.com, statement descriptor ≤ 22 chars (ej. AFILADOCS).
    • Representative: persona responsable (tú) — nombre legal, fecha de nacimiento, dirección residencial, DNI/NIE, relationship.representative=true.
    • Bank account (payouts): IBAN ES para cobros SEPA, titular coincidente con razón social. Stripe hace un pequeño depósito de verificación (0,01-0,99 €) que conviene ignorar, no hace falta confirmarlo en EU.
    • Verification documents: DNI/NIE anverso + reverso en PDF o JPG ≥ 1024×768. Si la entidad es S.L., añadir escritura de constitución + CIF. Stripe revisa en minutos-horas (típico: 5-30 min; máx. 1-2 días si dudas).
    Al finalizar cada bloque, el indicador "Activate payments" baja de rojo → amarillo → verde. Cuando está verde: curl -u sk_live_…: https://api.stripe.com/v1/account | jq '{charges_enabled,payouts_enabled,details_submitted}' debería devolver true, true, true. Hasta ese momento, Stripe permite crear sesiones en live pero al completar pago devuelve Your account cannot currently make live charges. Validar ANTES del smoke.
  3. 1Confirmar Vercel env: STRIPE_SECRET_KEY = sk_live_… (tras P0a) y NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY = pk_live_….
  4. 2Stripe Dashboard → Developers → Webhooks → crear endpoint https://afiladocs.com/api/webhooks/stripe con eventos checkout.session.completed, customer.subscription.{created,updated,deleted}, invoice.payment_failed.
  5. 3Copiar live signing secret (whsec_…, distinto del test) a Vercel env STRIPE_WEBHOOK_SECRET.
  6. 4Verificar EASYVERIFACTU_API_KEY apunta a producción (no sandbox) — primera factura real la usa.
  7. 5Redeploy y observar Sentry + Vercel logs en vivo.
  8. 6Smoke test: producto a 1 €, tarjeta personal real, checkout completo. Validar:
    • Pago visible en Stripe Dashboard (modo Live)
    • Webhook entregado con 200 en dashboard Stripe
    • orders.status='processing' + stripe_event_id en Supabase
    • Factura Verifactu generada (invoice_id poblado)
    • Email PaymentConfirmation recibido
  9. 7Reembolso del 1 € desde Stripe Dashboard → verificar email de refund + webhook de refund si está suscrito.
  10. ⚠Riesgo 1 (Verifactu): si la API key sigue en sandbox, el pedido queda sin factura. Rollback: borrar order manualmente y corregir key.
  11. ⚠Riesgo 2 (webhook): live whsec_ distinto al test. Si no se actualiza, las primeras firmas devuelven 400; Stripe reintenta 3 días pero ensucia logs. Validar primera entrega al instante.
  12. ⚠Riesgo 3 (idempotencia): confirmar que el índice UNIQUE sobre audit_log.stripe_event_id está aplicado en prod — si no, un reintento duplica efectos.

Riesgos transversales antes del go-live

  1. 1Verifactu live keys: la primera factura real revienta el smoke de P0b si sigue en sandbox. Validar EASYVERIFACTU_API_URL + _KEY antes del primer pago.
  2. 2Resend SPF/DKIM/DMARC: sin DKIM válido, PaymentConfirmation va a spam. Verificar dig TXT resend._domainkey.afiladocs.com + dig TXT _dmarc.afiladocs.com.
  3. 3Idempotencia webhooks — gap auditado 2026-04-18: la implementación actual (src/app/api/webhooks/stripe/route.ts:203-208) usa audit_log.findFirst({ where: { event: 'stripe_event.{id}' } }) antes de procesar. Best-effort pero no atómico: hueco de carrera si Stripe entrega dos retries en paralelo. audit_log.event NO tiene constraint UNIQUE (no se puede añadir sin romper otros eventos no-webhook como report.exported). Mitigación real: crear tabla dedicada processed_webhook_events(provider, event_id) con @@unique o columna orders.stripe_event_id @unique. Aceptable para launch (retries Stripe en minutos, no ms); mitigar post-launch como deuda. Abrir ticket de seguimiento.
  4. 4Uptime monitoring ausente: Sentry cubre errores de runtime, no caídas. Añadir UptimeRobot / Better Stack ping cada 5 min sobre /api/health → alerta a ops@ (tras P1).
  5. 5CSP en producción: monitorizar Sentry por violaciones de style-src/script-src 48 h tras el primer deploy live. style-src 'unsafe-inline' sigue activo por Tailwind v4 (aceptado).
  6. 6Backup Supabase: free tier sin PITR. Antes de datos reales, subir a Pro (25 $/mes, PITR 7 d) — coste casi obligado tras primer pago real.
  7. 7Páginas /legal coherentes con AEPD: tras P2, sincronizar /legal/privacidad, /legal/cookies y cookie banner con la lista final de encargados.
  8. 8Redirect www → non-www: confirmar curl -I https://www.afiladocs.com devuelve 301 a https://afiladocs.com.
  9. 9Rate-limit en prod: si UPSTASH_REDIS_REST_URL no está seteada, src/lib/rate-limit.ts:9 devuelve hasUpstash=false → todos los limiters son null → applyRateLimit responde {limited:false} y pasa todo. Crítico para /api/checkout y /api/contact — verificar antes del smoke.
  10. 10Región Vercel: ✅ resuelto en PR #26 — vercel.json declara "regions": ["cdg1"]. Corrección 2026-04-18: mad1 no existe en el catálogo Vercel (primera iteración del PR falló con "Invalid region"). cdg1 (París) es el PoP más cercano a ES — Vercel no tiene Madrid. Tras merge, próximo deploy desde París (−70-100 ms vs iad1).
  11. 11Env var Verifactu: código usa EASYVERIFACTU_* (sin guion bajo intermedio), no EASY_VERIFACTU_* como algunas docs indicaban. CLAUDE.md y guias/guia-workflows.md corregidos 2026-04-17. Verificar que el env var en Vercel tenga la clave correcta.
