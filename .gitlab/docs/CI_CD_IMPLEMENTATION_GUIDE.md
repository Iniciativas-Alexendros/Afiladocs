# Afiladocs DevSecOps CI/CD Implementation Guide

## Overview

This document outlines the architecture, security controls, and operational procedures for the Afiladocs Next.js 15 SaaS platform GitLab CI/CD pipeline. The pipeline adheres to industry standards, including NIST SP 800-204D, AWS CI/CD best practices, DoD Enterprise DevSecOps fundamentals, and SLSA Level 3 supply chain security targets.

## Pipeline Architecture

The pipeline uses GitLab's native DAG (Directed Acyclic Graph) features with `needs` relationships to maximize execution speed while enforcing strict quality and security gates.

### Stages

1. **Build**: 
   - Uses `node:22-alpine` for reproducible, hermetic builds.
   - Leverages `npm ci --prefer-offline` for deterministic dependency installation.
   - Caches the `.next/cache` and `.npm` folders to accelerate subsequent runs.
   - Outputs `.next/standalone`, `.next/static`, and `public` securely as immutable artifacts.

2. **Test**:
   - Executes static analysis (`test_lint`) and type checking (`test_types`).
   - Runs in parallel once the `build` phase stabilizes its cache.
   - Fails the pipeline if strict TypeScript or ESLint rules are violated.

3. **Security**:
   - Integrates native GitLab Ultimate/Premium templates:
     - `SAST` (Static Application Security Testing)
     - `Secret-Detection`
     - `Dependency-Scanning`
   - Scans against OWASP Top 10 vulnerabilities, leaked credentials, and CVEs in npm dependencies.

4. **Deploy**:
   - Includes `deploy_staging` and `deploy_production` jobs.
   - Implements safe Rsync over SSH using keys passed via masked CI/CD variables.
   - Prevents Man-in-the-Middle (MitM) attacks by injecting strictly verified `known_hosts` keys.
   - Employs zero-downtime reloads using PM2 cluster mode.
   - Utilizes `needs:optional` for security jobs (e.g., `sast`, `secret_detection`) to prevent pipeline DAG parsing errors when conditional security jobs are not generated by the rules engine.

## Security Control Mapping Matrix

| Security Goal | Implementation in Pipeline | Standard/Framework |
|---------------|----------------------------|--------------------|
| **Hermetic Builds** | Hardcoded Node version (`node:22-alpine`), locked dependencies (`npm ci`) | SLSA Level 3 |
| **Secrets Management** | Vault/GitLab variables masked & protected. Passed in memory. | NIST SP 800-204D |
| **Vulnerability Scanning** | SAST, Secret Detection, Dependency Scanning | OWASP / DoD DevSecOps |
| **Immutable Artifacts** | Artifacts strictly limited to `standalone`, `static`, `public` | SLSA Level 2 |
| **Least Privilege / RBAC** | Staging and Prod deploy uses separate SSH keys (`$SSH_PRIVATE_KEY_STAGING`, `$SSH_PRIVATE_KEY_PROD`) | AWS Security Pillar |
| **Audit Trails** | Manual approval gates for staging and production deployments | SOC 2 / HIPAA |

## Operational Procedures & Prerequisites

### GitLab Project Settings

To ensure the pipeline functions securely, configure the following settings in GitLab (**Settings > Repository** & **Settings > CI/CD**):

1. **Protected Branches**:
   - Protect the `main` or `master` branch.
   - Require approval from Code Owners before merging.
   - Block forced pushes and branch deletion.

2. **Merge Request Approvals**:
   - Require at least 1 approval.
   - Prevent the author from approving their own MR.
   - Enable "Require new approvals if new commits are added".

3. **Variables (Settings > CI/CD > Variables)**:
   Ensure all below variables are marked as **Masked** and **Protected** (where applicable):

   | Variable | Description |
   |----------|-------------|
   | `SSH_PRIVATE_KEY_STAGING` | Ed25519/RSA private key for staging server access. |
   | `SSH_KNOWN_HOSTS_STAGING` | The public host key (`ssh-keyscan`) of the staging server. |
   | `DEPLOY_USER_STAGING` | SSH username for staging (e.g., `deployer`). |
   | `DEPLOY_HOST_STAGING` | IP or hostname of the staging server. |
   | `DEPLOY_PATH_STAGING` | Absolute deployment path on the staging server. |
   | `SSH_PRIVATE_KEY_PROD` | Ed25519/RSA private key for production server access. |
   | `SSH_KNOWN_HOSTS_PROD` | The public host key of the production server. |
   | `DEPLOY_USER_PROD` | SSH username for production. |
   | `DEPLOY_HOST_PROD` | IP or hostname of the production server. |
   | `DEPLOY_PATH_PROD` | Absolute deployment path on the production server. |

### Environment Variable Templates (.env.example)

Review the `env.ts` and `.env.example` in the repository for environment-specific secrets that need to be injected either via GitLab CI variables into `.env.production` at build time or directly stored securely on the deployment VPS.

### Troubleshooting

- **Error: `job needs 'sast' job, but 'sast' does not exist`**:
  - *Cause*: A deployment job depended strictly on `sast` via the `needs` array, but GitLab skipped creating the `sast` job due to conditional rule evaluation.
  - *Fix Applied*: We updated the DAG to use `{ job: sast, optional: true }`. Ensure any future optional jobs follow this format.
- **Rsync / SSH Connection Refused**:
  - *Cause*: Invalid `SSH_KNOWN_HOSTS_*` value or network firewall blocking the GitLab runner IP.
  - *Fix*: Update the known hosts variable with the latest output of `ssh-keyscan -H <your-server-ip>`.

## Maintenance Schedule

- **Monthly**: Review CI/CD pipeline template updates from GitLab Ultimate.
- **Quarterly**: Rotate SSH deploy keys (`SSH_PRIVATE_KEY_STAGING`, `SSH_PRIVATE_KEY_PROD`).
- **Continuous**: Monitor security dashboard for Dependency Scanning CVEs and apply `npm audit fix` accordingly.
